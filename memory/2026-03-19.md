## Watchdog: Duplicate Alert Spam Fix (Session 3)

### Problem
- After deploying inotify + completion fixes, user reported "20 alerts" spam
- Root cause: `delete-telegram-msgs.sh` was forwarding messages to read their content, and the log tailer was interpreting every `sendMessage ok` log entry as a new alert
- Secondary bug: `completed_fast` runs were excluded from state persistence → re-detected as "new" on next cycle → repeated spawn+completion alerts
- Tertiary bug: `completed` flag in state was being overwritten during state persistence

### Fix
- Persisted `completed_fast` runs in state file with `completed: true` flag
- State persistence now preserves existing `completed` flag on rewrites
- Killed `delete-telegram-msgs.sh` processes (actual source of the 20-message burst)
- Removed debug logging from watchdog (alert-debug.log entries)
- Verified clean operation with test run `92644329-a729-4d70-8c30-4429caef302a`

### Key Lessons
- Watchdog itself was working fine — the spam was from an unrelated script
- Always check for other processes touching the same APIs when debugging alert storms
- `completed_fast` runs MUST stay in state or they'll loop forever

## Watchdog: Sub-Agent Alert Fixes (Session 2)

### Problem
- Sub-agent alerts were hit-and-miss (polling every 30s missed fast runs)
- Completion alerts only showed run ID — no agent, model, task info
- Two competing detection paths (log tailer + runs.json polling) could double-fire or both miss

### Fix: Event-Driven Detection
- Installed `inotify-tools` — `inotifywait` watches `runs.json` for filesystem events
- New `start_runs_watcher()` function: instant detection on modify/close_write
- Runs.json polling removed from main loop (inotify is primary, 60s timeout as safety fallback)

### Fix: Full Details in State File
- State file (`last-runs-state.json`) now stores: agent, model, task, startedAt
- Completion alerts look up stored details → same info as spawn alerts
- JSON-escaped task field to handle special chars

### Fix: Deduplicated Detection Paths
- Log tailer: only handles **error** completions (`isError=true`)
- `check_new_subagents()`: owns **success** completions with full stored context
- No more double-fire risk
